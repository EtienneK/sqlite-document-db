# Decisions to make first

Two open questions gate a lot of the work below. Both are **much cheaper to decide now
than later**: 2.0.0 has not been published, so the storage format and the API shape are
still free to change. After a release, either becomes a migration problem.

## DR-1: Document storage format

**Status: DECIDED 2026-07-22 — option C. IMPLEMENTED same day.** Dates are stored in
EJSON format (`{"$date": "..."}`) via [src/ejson.ts](src/ejson.ts) (~100 lines, zero
dependencies kept); other non-JSON types are rejected at write time with the offending
path. Query support: `$eq/$ne/$gt/$gte/$lt/$lte` target the `field.$date` sub-path,
`$in/$nin` with Dates rewrite to `$or/$nor` of equalities, whole-object/array equality
encodes query values through the same encoder. Verified dual-engine in
[test/dates.spec.ts](test/dates.spec.ts). Not yet date-aware: `$all` and `$elemMatch`
on date elements (niche; revisit with item 3). Full EJSON (option B) remains open —
the wire format is identical. Gates items [2](#2-createindex-and-friends), [3](#3-implicit-array-element-matching),
[5b](#5-typescript-typing) and [6](#6-cursor-sort-limit-and-skip), and caps DR-2.

### Context

Documents are stored with `JSON.stringify` into a `data JSON` column. JSON has no
`Date`, `ObjectId`, `Binary`, `Decimal128`, `Long` or `RegExp` — so those types cannot
survive a round trip.

This is not hypothetical. Measured 2026-07, inserting the same document into both
engines and reading it back:

| field | sqlite-document-db | mongodb |
| --- | --- | --- |
| `date` | **`String`** `"2020-01-02T03:04:05.000Z"` | `Date` |
| `regex` | **`Object {}`** — silently destroyed | `RegExp` |
| `bytes` (`Uint8Array`) | **`Object {"0":1,"1":2,"2":3}`** | `Binary` |
| `nested.when` | **`String`** | `Date` |
| `missing: undefined` | key absent | `null` |

And the consequence that actually bites:

```
findOne({ date: <the Date that was just stored> })
  sqlite-document-db → NOT FOUND
  mongodb            → FOUND
```

Dates degrade to strings; regexes and binary data are silently corrupted. Almost every
real application stores dates, so this is the single largest barrier to being usable as
a MongoDB substitute — and no amount of type-level work fixes it, because the types
would be *correct* about a schema the storage layer cannot hold.

### Options

**A. Keep plain JSON.** Document the supported types and reject (or warn on) the
others at insert time rather than corrupting them silently. Cheapest; caps DR-2 at
"compatible subset" forever.

**B. Adopt [Extended JSON](https://www.mongodb.com/docs/manual/reference/mongodb-extended-json/) (EJSON).**
MongoDB specified EJSON for exactly this problem and the `bson` package implements it.
`{ when: new Date() }` becomes `{"when":{"$date":"…"}}` and round-trips faithfully.

**C. Middle path — EJSON encoding for `Date` only** (and `ObjectId` later), using the
real EJSON wire format so it is forward-compatible with option B. Fixes the case
everyone actually hits without committing to the full type system.

### Consequences of B (and, in miniature, C)

- **[src/query/query.ts](src/query/query.ts) gets harder.** `json_extract(data,'$.when')`
  returns an *object*, not a comparable scalar. The compiler must know a field holds a
  wrapped type and target `$.when.$date` instead. There is no schema, so it has to be
  inferred from the *query value's* runtime type — `{ when: someDate }` compiles
  differently from `{ when: someString }`.
- **Indexes must target the identical sub-path** or they silently stop matching. Item 2
  already insists index paths and query paths be generated by the same code; this makes
  that non-negotiable.
- **Sort order** (item 6) must follow BSON comparison order, not SQLite's.
- **Zero dependencies would be lost** if `bson` is pulled in. A minimal EJSON subset for
  `Date`/`ObjectId`/`Binary` is on the order of 100 lines and would preserve it — worth
  it given that zero-dep is now a headline feature.
- **Storage grows**; every wrapped value becomes an object.

### Recommendation

**C**, then reassess. It removes the failure everyone hits, uses the real EJSON format
so option B stays open, and doesn't spend the zero-dependency property. Whichever is
chosen, do it before publishing 2.0.0 and state the supported type list in the README —
silent corruption of `RegExp` and `Uint8Array` is worth fixing even under option A.

## DR-2: How MongoDB-compatible should the API be?

**Status: DECIDED 2026-07-22 — option 2, "compatible subset".** Method names,
signatures and result shapes match the official driver for the CRUD surface;
deviations get documented. Types are vendored from the driver (Apache-2.0, with
attribution), never imported from it. Shapes items [5b](#5-typescript-typing), [15](#15-remaining-collection--db-api)
and [16](#16-aggregation-pipeline).

### Options

1. **"Inspired by MongoDB"** — the status quo. Free to diverge wherever SQLite suggests
   something better.
2. **"Compatible subset"** — method names, signatures and result shapes match the driver
   for the CRUD surface; deviations documented. Application code that sticks to CRUD
   ports by changing one import.
3. **"Drop-in replacement"** — requires DR-1 option B *and* aggregation (item 16) *and*
   BSON types. Realistically a different, much larger project.

### What was established about feasibility

Checked against the driver's published types (`mongodb@7`, 9,076 lines, 389 exports):

- **No nominal barrier.** `Collection<TSchema>`'s internals are stripped from the
  published `.d.ts` (`/* Excluded from this release type: s */`), leaving no
  `private`/`#` members. Structural conformance is therefore possible.
- **But don't chase type-level assignability.** Being assignable to
  `mongodb.Collection` is all-or-nothing: ~35 methods plus a `readonly db: Db` that
  recursively drags in their `Db` class. Target *source-level* compatibility — app code
  compiles and runs after swapping the import — which only needs agreement on the
  methods people actually call.
- **`_id` as a string is not disqualifying.** Their `InferIdType<TSchema>` only defaults
  to `ObjectId` when the schema doesn't declare `_id`; a user writing
  `interface User { _id: string }` type-checks fine.
- **Convergence has already happened by accident.** This library's `WithoutId<TSchema>`
  is identical to the driver's.

### Gotchas

- **Don't adopt their `Filter` before items 3 and 8 land.** Their
  `AlternativeType<T> = T extends ReadonlyArray<infer U> ? T | RegExpOrString<U> : RegExpOrString<T>`
  already promises implicit array-element matching *and* regex values. Types that
  over-promise are worse than weak types.
- **Result shapes differ in small, load-bearing ways.** `InsertOneResult` is missing
  `acknowledged`; `UpdateResult` is missing `matchedCount` and `upsertedId`. These are
  exactly the fields real code destructures.
- **Vendor the types, don't depend on `mongodb`.** An `import type { Filter } from 'mongodb'`
  in the published `.d.ts` forces every TypeScript consumer to install the driver just
  to typecheck, destroying the zero-dependency property. The driver is Apache-2.0, so
  copying the relevant declarations with attribution is permitted.
- **Aggregation is the swing factor.** "80% of typical CRUD code" is reachable without
  it; "80% of real applications" probably isn't, since many use `aggregate()`.

### Testing leverage

The dual-engine suite **is already the conformance suite**. If the APIs converge, the
`byId()` helper and the `Db | Mdb` casts in [test/index.spec.ts](test/index.spec.ts)
become unnecessary — and the helper's accessor could be typed as the *driver's*
`Collection`, at which point any API divergence fails at compile time instead of
becoming a bug report.

### Recommendation

**Option 2.** Honest, achievable, and it makes the existing test architecture pay off.
Say so in the README — "MongoDB-compatible API for the CRUD subset", not "drop-in
replacement".

## 1. Rework the cursor off rowid pagination

**Size: S — DONE 2026-07-22.** `find()` now prepares one statement per cursor and
streams it with `StatementSync.iterate()`, with no `ORDER BY` (order without `sort()`
is unspecified, matching MongoDB — a bare `ORDER BY rowid` was measured to defeat
field indexes too). Added `cursor.close()`, statement release on `for-await` break,
[test/cursor.spec.ts](test/cursor.spec.ts), and a plan-regression guard in
[test/query-plan.spec.ts](test/query-plan.spec.ts) that fails if `find()`'s SQL ever
stops being index-eligible. One trap for posterity: the parens around the compiled
filter are load-bearing — `$exists` emits a bare scalar subquery. Original analysis
follows.

`find()` currently re-runs a `LIMIT 1` query for every single document:

```sql
SELECT rowid, data FROM collection_x WHERE rowid > ? AND (<filter>) ORDER BY rowid LIMIT 1
```

Two problems:

1. **It re-prepares the statement on every `next()`** ([src/index.ts](src/index.ts), in
   `find`). One `prepare()` per row.
2. **It defeats any index on the filtered field.** Verified with `EXPLAIN QUERY PLAN`
   against 20k rows and an index on `json_extract(data, '$.qty')`:

   | Query | Plan |
   | --- | --- |
   | `WHERE json_extract("data",'$.qty') > 100` | `SEARCH USING INDEX ix_qty` |
   | `WHERE rowid > ? AND json_extract("data",'$.qty') > 100 ORDER BY rowid LIMIT 1` | `SEARCH USING INTEGER PRIMARY KEY (rowid>?)` |

   The `ORDER BY rowid` makes SQLite walk the table in rowid order and filter as it
   goes, so item 2 below buys nothing until this is fixed.

**Approach.** Prepare the statement once per cursor and stream it with
`StatementSync.iterate()` (available in `node:sqlite`), dropping the `rowid > ?`
predicate and the `ORDER BY` entirely. Keep `next()`/`toArray()`/async-iteration as
the public surface so nothing downstream changes.

Watch out for: `toArray()` must still work after partial iteration, and the cursor
should hold its statement open until exhausted or closed — consider adding
`cursor.close()`, which MongoDB also has.

**Verify.** Existing suite stays green; add an `EXPLAIN QUERY PLAN` assertion (see
item 2) proving a filtered `find()` uses an index.

## 2. `createIndex()` and friends

**Size: M — DONE 2026-07-22.** `createIndex()` (single, compound, unique, custom
name), `dropIndex()`, `indexes()` and `listIndexes()`, with MongoDB-style generated
names, all verified dual-engine in [test/indexes.spec.ts](test/indexes.spec.ts).
Index paths are built by the exported `toJson1PathString()` — the same code queries
use. Because Dates are stored wrapped (DR-1), single-field indexes get a non-unique
companion index on `<field>.$date`; [test/query-plan.spec.ts](test/query-plan.spec.ts)
replays the *captured* CREATE INDEX + SELECT statements and asserts via
`EXPLAIN QUERY PLAN` that both the plain and the date index actually serve `find()`.
Not done: compound-index date companions (niche), `dropIndexes()`/`indexExists()`,
index direction only matters once item 6 (sort) lands. Original analysis follows.

*This is the feature that justifies building on SQLite at all* — it's
what a hand-rolled JSON-file document store can't do.

Collections currently have exactly one index, the unique `_id` index created in the
`Collection` constructor. Every other query is a full scan.

The good news, already verified: **SQLite expression indexes work with the SQL this
library generates today, unchanged.** An index created as
`CREATE INDEX ix ON collection_items(json_extract(data, '$.qty'))` is used by both
`$gt` and `$eq` — even though `$eq` compiles to `is` rather than `=`. No changes to
[src/query/query.ts](src/query/query.ts) are needed.

**Approach.**

```javascript
await db.collection('items').createIndex({ qty: 1 })
await db.collection('items').createIndex({ 'size.uom': 1, status: -1 }, { unique: true })
await db.collection('items').listIndexes()
await db.collection('items').dropIndex('qty_1')
```

- Map each key to `json_extract(data, '$.<path>')`, reusing `toJson1PathString()`
  from [src/query/query.ts](src/query/query.ts) so index paths and query paths are
  generated by the *same* code — otherwise they will drift and silently stop matching.
- Name indexes the way MongoDB does (`qty_1`, `size.uom_1_status_-1`) so
  `dropIndex(name)` is compatible.
- SQLite ignores index *direction* for single-column lookups but honours it for
  multi-column ordering; store it for when item 6 lands.
- `listIndexes()` can read back from `sqlite_master`.

**Non-obvious risks.**
- **Settle [DR-1](#dr-1-document-storage-format) first.** If wrapped types land later,
  every index built over a wrapped field targets the wrong path and silently stops
  matching — with no error, just a return to full scans.
- Sparse/partial semantics differ: a missing field yields SQL `NULL`, so a `unique`
  index rejects a *second* document missing that field. MongoDB's non-sparse unique
  index behaves the same way, so verify against MongoDB rather than guessing.
- Item 3 (implicit array matching) may compile to `OR EXISTS(...)`, which typically
  can't use an index. Decide then whether to emit a different plan when an index
  exists, or accept the scan.

**Verify.** Assert on `EXPLAIN QUERY PLAN` output — a test helper that fails when a
query that *should* hit an index reports `SCAN` is the only way to stop this
regressing. Pair with benchmarks (item 17).

## 3. Implicit array element matching

**Size: M — DONE 2026-07-22.** `{ tags: 'B' }` and the comparison/`$in`/`$nin`
operators now match array elements like MongoDB; all four disabled assertions are
enabled and the tutorial's query-an-array examples added, dual-engine verified.
Implementation notes for posterity: a flat `OR` is NEVER indexed (SQLite's
OR-optimization does not apply to expression indexes at all), so top-level
comparisons compile to `rowid IN (scalar-arm UNION ALL element-arm)`, where the
element arm leads with `extract >= '[' AND extract < '\'` — JSON arrays extract as
text starting with `[`, so this range probes THE SAME expression index (0.45-0.73ms
vs 9ms scans on 20k rows; `UNION ALL` not `UNION`, whose dedup sort blocked arm
indexing; `IN` dedups anyway). Range operators gained MongoDB-style type bracketing
(a number query no longer matches text/arrays/objects — SQLite's type ordering
otherwise made `> 25` true for every array). `ORDER BY rowid` is restored in
`find()`/`findOne()` — safe now that index work happens inside the rowid subqueries.
Still open: implicit matching inside `$elemMatch` nesting uses the unindexed flat
form (fine), `$all` on dates. Original analysis follows. it single-handedly unblocks all four
commented-out assertions in
[test/operators/query-operators.spec.ts](test/operators/query-operators.spec.ts#L29-L34).

In MongoDB, `{ tags: 'B' }` matches a document whose `tags` is `'B'` **or** an array
*containing* `'B'`. This library only does the former. The same rule explains the
second disabled pair: `{ tags: ['A','B'] }` matches both an exact array equal to
`['A','B']` and an array that *contains* `['A','B']` as an element.

Referenced by the README as
[Query an Array for an Element](https://www.mongodb.com/docs/manual/tutorial/query-arrays/#query-an-array-for-an-element).

**Approach.** Equality (and `$in`, `$gt`, … — MongoDB applies this to all of them)
becomes "scalar match OR element match":

```sql
json_extract(data,'$.tags') = <v>
OR EXISTS (SELECT 1 FROM json_each(data,'$.tags') WHERE value = <v>)
```

Guard the `json_each` against non-array values with `json_type(...) = 'array'`;
`json_each` on a scalar yields one row and will otherwise produce false positives.

Note this compiles to `OR EXISTS(...)`, which typically cannot use an index — so it
pulls against item 2, and both are sensitive to [DR-1](#dr-1-document-storage-format)
because element comparison happens on whatever the stored representation is.

**Verify.** Uncomment the four assertions and make them pass on both engines. Then
audit the other operators — this rule is why `$ne` on an array field is famously
surprising in MongoDB, and the parity tests will catch it.

## 4. `updateOne` / `updateMany` with update operators

**Size: M — DONE 2026-07-22** for the core: `updateOne`/`updateMany` with
`$set`/`$unset`/`$inc` (composable in one statement via nested
`json_set`/`json_remove`; dotted `$set`/`$inc` paths create missing parents with
`json_insert`, which no-ops when present). No-op updates report `modifiedCount: 0`
via `AND data != <new-expr>` in the UPDATE. All result shapes now match the driver
(`acknowledged`/`matchedCount`/`modifiedCount`/`upsertedCount`/`upsertedId`, and
`acknowledged` on insert/delete results too — DR-2). `replaceOne` rejects operator
documents; update methods reject operator-less documents. Dual-engine verified in
[test/mdb-tutorials/update-documents.spec.ts](test/mdb-tutorials/update-documents.spec.ts).
**Still open:** `upsert` option, `$mul`/`$min`/`$max`/`$rename`/`$push`/`$pull`/
`$addToSet`/`$pop`, `findOneAndUpdate`/`findOneAndReplace`/`findOneAndDelete`.
Known divergences (documented, not tested): `$inc` on a non-numeric field coerces
instead of erroring; `$unset` of an array element removes it instead of nulling it.
Original analysis follows. only `replaceOne` exists, so there is no way
to modify a field without rewriting the whole document.
From the README, and the
[Update Documents tutorial](https://www.mongodb.com/docs/manual/tutorial/update-documents/).
The Postgres project supports `$set` and `$inc`.

**Approach.** SQLite's JSON functions map onto the update operators cleanly:

| Mongo | SQLite |
| --- | --- |
| `$set` | `json_set(data, '$.path', value)` |
| `$unset` | `json_remove(data, '$.path')` |
| `$inc` / `$mul` | `json_set(data, '$.p', json_extract(data,'$.p') + n)` |
| `$rename` | `json_set` + `json_remove` |
| `$push` / `$addToSet` | `json_insert` into `$.arr[#]` |
| `$pop` / `$pull` | rebuild via `json_group_array` + `json_each` |
| `$min` / `$max` | `json_set` guarded by a `CASE` |

Operators compose in one statement by nesting `json_set` calls. Do `$set`/`$unset`/
`$inc` first — they cover most usage and match the Postgres project's scope.

Also add the `upsert` option, and `findOneAndUpdate` / `findOneAndReplace` /
`findOneAndDelete` once updates exist.

**Verify.** Extend
[test/mdb-tutorials/update-documents.spec.ts](test/mdb-tutorials/update-documents.spec.ts),
which currently only tests `replaceOne`.

## 5. TypeScript typing

This is a TypeScript-first library, but the type surface doesn't currently deliver on
that. Split into a quick fix and a real project.

### 5a. Make the generics reachable — **Size: S — DONE 2026-07-22**

`Collection<TSchema>` is generic and `TSchema` correctly threads through `find()`,
`findOne()` and `insertOne()`. But `Db.collection()` is declared as:

```typescript
collection (name: string): Collection        // = Collection<Document>
```

It returns the *default* instantiation, so `db.collection<User>('users')` does not
compile and **every generic in `Collection` is unreachable from the public entry
point.** All that machinery is currently dead weight.

```typescript
collection <TSchema extends Document = Document>(name: string): Collection<TSchema>
```

The internal cache is `{ [key: string]: Collection }`, so this needs a cast at the
cache boundary — the type parameter is a caller-side assertion about a collection's
contents, which is exactly how the MongoDB driver treats it too. Worth a comment
saying so, and worth deciding what happens if two callers request the same collection
at different types.

### 5b. Typed filters and results — **Size: M, best done after items 4, 6 and 7**

> Depends on both decision records. [DR-2](#dr-2-how-mongodb-compatible-should-the-api-be)
> decides whether these types are written from scratch or adopted from the driver;
> [DR-1](#dr-1-document-storage-format) decides whether `Date` is even expressible in a
> schema. Writing this layer before both are settled means writing it twice.

`Filter` is `Record<string, any>`, so `find({ qtyy: { $lt: 30 } })` compiles happily
despite the typo, and `{ $lt: 'thirty' }` isn't caught either. Worth building:

- **`Filter<TSchema>`** — operators typed against the field they apply to, so `$lt`
  wants the field's type and `$in` wants an array of it. This is the big win.
- **Dot-notation paths** — template literal types can type `'size.uom'` against a
  nested schema, and `` `${K}.${number}` `` for array indexing. Genuinely tricky at
  depth; cap recursion or accept `string` as a fallback.
- **Operator-specific value types** — `$exists: boolean`, `$size: number`,
  `$all: T[]`, `$elemMatch` against the element type. `convertOp` already validates
  these *at runtime* and throws, so the type layer is encoding rules that already
  exist — a good sign the design is sound.
- **Projection changes the result type** (item 7): `{ projection: { item: 1 } }`
  should narrow the returned document. Needs item 7 to exist first.
- **`UpdateFilter<TSchema>`** for item 4's operators. Needs item 4 first.

**Sequencing:** do 5a now, hold 5b until the API stops moving — typing an API that's
about to grow `sort`, `projection` and `update` means doing it twice.

**Verify.** Types need their own tests; the runtime suite won't catch regressions.
Vitest supports `expectTypeOf` assertions (`vitest --typecheck`), which is the natural
fit. Include *negative* cases — assert that a misspelled field or wrong operator value
**fails** to compile, since that's the actual feature.

Related: the `byId()` helper and `as any` casts in
[test/index.spec.ts](test/index.spec.ts) exist because this library types `_id` as
`string` while the MongoDB driver types it as `ObjectId`. If `_id` typing is revisited
(item 17), those casts may be removable.

## 6. Cursor `sort()`, `limit()` and `skip()`

**Size: M — DONE 2026-07-22.** Chainable `sort()`/`limit()`/`skip()` plus the
`find(filter, { sort, limit, skip })` options form, throwing once iteration starts
(like the driver). Sorting emits a type-rank CASE per key emulating MongoDB's BSON
comparison order — null/missing < numbers < strings < booleans < dates — with
wrapped `{"$date"}` values ranked as dates and compared by ISO string; verified
dual-engine in [test/sort-limit-skip.spec.ts](test/sort-limit-skip.spec.ts)
including mixed-type fields. `limit(0)` = no limit; ties break by rowid (natural
order). **Known divergence:** MongoDB sorts array fields by min (asc) / max (desc)
element; here arrays rank as a group and compare as text. **Not done:** sorting via
an index (the rank CASE always filesorts — revisit with benchmarks, item 17).
Original analysis follows.

**Approach.** `ORDER BY json_extract(data,'$.f') ASC/DESC`, `LIMIT`, `OFFSET`.
The interesting part is **type ordering**: SQLite sorts NULL < number < text < blob,
whereas MongoDB has its own [BSON comparison order](https://www.mongodb.com/docs/manual/reference/bson-type-comparison-order/).
Mixed-type fields will disagree — decide whether to emulate MongoDB (a `CASE` on
`json_type` to produce a type-rank sort key) or document the difference. The parity
tests will surface this immediately.

Note this interacts with item 1: sorting by an indexed field should use the index
rather than a filesort.

## 7. Projection

**Size: M — DONE 2026-07-22.** `find(query, { projection })`, the chainable
`cursor.project()`, and `findOne(filter, { projection })` (findOne now delegates to
`find(..., { limit: 1 })`, gaining sort support for free). Inclusion, exclusion, the
`_id` exemption, dotted paths, and projecting INTO arrays of embedded documents —
all dual-engine verified in
[test/mdb-tutorials/project-fields.spec.ts](test/mdb-tutorials/project-fields.spec.ts).
Deliberately applied **in JS to decoded documents, not in SQL**: `json_object`
reconstruction cannot distinguish missing fields from null ones (MongoDB omits
missing), and array-of-documents projection is trivial in JS. Invalid projections
(inclusion/exclusion mixes, path collisions) throw at iteration time, where the
driver reports them. **Deferred:** `$slice`, `$elemMatch`, `$` positional.
Original analysis follows (note it proposed the SQL-side approach — superseded).

## 8. Remaining query operators

**Size: M — DONE 2026-07-22** for the headline three, closing the parity gap with the
Postgres project. `$regex` (string, RegExp and `$options` forms; implicit
`{ field: /re/ }`; regexes inside `$in`/`$nin`/`$not`/`$elemMatch`; array element
matching), `$type` (name and numeric aliases, arrays of aliases) and `$mod`, all
dual-engine verified in
[test/operators/evaluation-operators.spec.ts](test/operators/evaluation-operators.spec.ts).
Notes for posterity:

- `$regex` compiles to a `mdb_regexp()` SQL function that `Db.fromUrl` registers on
  the connection (JS `RegExp`-backed, per-connection pattern cache). This **raised the
  Node floor to 22.13** (`DatabaseSync.prototype.function`). MongoDB's `x` option has
  no JS equivalent and throws; `g`/`y` are stripped (stateful `test()` would skip rows).
- An **explicit** `$eq` against a regex compiles to `FALSE` — MongoDB only
  equality-matches *stored* regex values, which cannot exist here — while `$ne`/ranges
  throw, matching the server. `$options` without `$regex` throws; the pair must reach
  the compiler as ONE operator (see the `convert()` preprocess), or the
  multiple-operator branch splits them.
- `$type`'s number aliases mirror the driver's serialization: integral JS numbers in
  int32 range are `'int'`, everything else `'double'` — so both aliases carry int32
  range brackets. Unstorable BSON types (`'long'`, `'binData'`, …) are valid aliases
  that match nothing; unknown aliases throw. The stored-Date wrapper ranks as
  `'date'`, never `'object'`.
- `$mod` truncates decimal arguments AND decimal field values toward zero
  (oracle-verified), which SQLite's `CAST`/`%` do natively.
- Fixed in passing, latent `$elemMatch` crashes ("malformed JSON"): on scalar
  fields, and on string array elements. It now uses 2-arg `json_each(doc, path)` +
  `json_quote(value)` and only matches real arrays. (`$all` still has the same
  scalar-field hazard — it feeds `json_each` an extracted value.)

Still open below: `$expr`, `$bits*`, `$text`; `$where` is a documented won't-do.
Original analysis follows.

- **`$regex`** — SQLite has no built-in `REGEXP`, but `node:sqlite` lets you register
  one: `db.function('regexp', (pattern, value) => ...)` backed by JavaScript `RegExp`.
  Mind the argument order (SQLite calls `regexp(pattern, value)` for `value REGEXP pattern`)
  and support the `i`/`m`/`s` option flags. Unblocks three `TODO`s in the operator spec.
- **`$type`** — maps to `json_type(data, '$.path')`, but MongoDB's type names/numbers
  (`'double'`, `'string'`, `16`, `'long'`, …) don't line up with SQLite's five JSON
  types. Needs an explicit mapping table and a documented list of types that can't be
  represented. Referenced by the README.
- **`$mod`** — `json_extract(...) % divisor = remainder`. Trivial once the operator
  table is being extended.
- **`$expr`** — lets a query compare two fields of the same document. Referenced by two
  `TODO`s in the operator spec. Larger than it looks: it needs an aggregation-expression
  evaluator, so it may belong with item 16.
- **`$bitsAllSet` / `$bitsAnySet` / `$bitsAllClear` / `$bitsAnyClear`** — SQLite has
  `&`/`|`. Low demand; listed as "todo" in the Postgres project too.
- **`$text`** — would need an FTS5 virtual table shadowing the collection. A genuinely
  interesting stretch goal, and something the Postgres project doesn't do.
- **`$where`** — **won't do.** Executes arbitrary JavaScript; the Postgres project
  explicitly lists it as impossible. Document it as unsupported.

Also: `$in`/`$nin` against array fields (two `TODO`s) falls out of item 3.

## 9. Use bound parameters instead of string interpolation

**Size: M.** [src/query/query.ts](src/query/query.ts) builds SQL by concatenation,
escaping values through `quote()` (doubling `'`) and identifiers through `quote2()`.
That is *correct* as written, and collection names are regex-validated — this is
hardening, not an open hole.

**Why it's worth doing anyway:** every distinct filter *value* produces a distinct SQL
string, so prepared statements can never be reused or cached; and the safety of the
whole thing rests on every future operator remembering to call `quote()`.

**Approach.** Have `convert()` return `{ sql, params }` with `?` placeholders and
thread the parameter array through. This is a mechanical but wide refactor of
`convertOp`. Do it *before* the operator set grows much further (items 3, 4, 8),
because each new operator makes it more expensive.

**Verify.** The existing suite is a strong safety net here — it exercises every
operator against real MongoDB. Add a test with adversarial values (`'`, `"`, `\`,
`$`, `--`, embedded JSON, dotted field names).

## 10. Normalise errors to MongoDB shapes

**Size: S.** A duplicate `_id` currently surfaces as a raw SQLite
`SQLITE_CONSTRAINT_UNIQUE` error. MongoDB throws a `MongoServerError` with
`code: 11000`, and callers routinely branch on that code.

The existing test only asserts `expect(error).toBeDefined()`
([test/index.spec.ts](test/index.spec.ts)), which is why this was never noticed.

**Approach.** A small error-mapping layer plus exported error classes; tighten the
existing assertions to check the code on both engines.

## 11. Fix collection naming restrictions

**Size: S.** `Db.collection()` lowercases the name, and the `Collection` constructor
rejects anything not matching `/^[a-z_]+[a-z0-9_]*$/`.

So `db.collection('Users')` and `db.collection('users')` **silently return the same
collection**, and `db.collection('my-data')` throws. MongoDB collection names are
case-sensitive and permit far more.

**Approach.** The restriction exists because the name is interpolated into SQL. Quote
the identifier properly instead (`quote2()` already does this for columns) and drop the
lowercasing. Keep a much narrower rejection list (empty, `$`, NUL, leading `sqlite_`).
Note this changes behaviour for anyone relying on the lowercasing — worth a note in
the 2.x release notes.

## 12. Transactions

**Size: M.** `insertMany` is deliberately non-transactional to match MongoDB's *ordered*
insert semantics, and that should stay. But there's currently no way for a *caller* to
group operations atomically.

`node:sqlite` supports this directly (`BEGIN`/`COMMIT` via `exec`, plus
`DatabaseSync.isTransaction`). Expose something like
`await db.withTransaction(async () => { ... })`, closer to SQLite's strengths than
MongoDB's session API.

While here, set sensible pragmas for file-backed databases — `busy_timeout` especially,
since WAL mode still blocks on a writer. Currently only `journal_mode=WAL` is set, which
is a no-op for `:memory:` databases.

## 16. Aggregation pipeline

**Size: L.** `$match`, `$group`, `$sort`, `$project`, `$limit`, `$unwind`, `$lookup`.

Genuinely feasible — `$unwind` is `json_each`, `$lookup` is a join, `$group` is
`GROUP BY`, and `$match` reuses the existing compiler — but it is comfortably the
biggest item here and arguably a different project.

**Decide explicitly whether this is in scope**, and say so in the README either way.
"Query and CRUD only, no aggregation" is a perfectly respectable position that saves
people from filing issues about it. The Postgres project doesn't do aggregation either.

## 17. Smaller items and nice-to-haves

- **Benchmarks.** There's no way to demonstrate that item 2 helped. A small
  insert/query benchmark over ~100k documents, ideally in CI to catch regressions.
- **`_id` types.** Tests use numeric `_id`s and they work, but this isn't specified
  anywhere. Decide what's supported, type it (item 5b), and document it.
- **Statement cache.** Once item 9 lands, cache prepared statements per collection.
- **Document size / depth limits.** SQLite has its own limits; find out what they are
  and document them rather than letting users discover them.
- **API documentation.** TypeDoc from the source, published to GitHub Pages.
- **Remove `backup/`.** Dead code from the original implementation, kept in git
  history. `git rm -r backup/` — this was attempted during the 2026 modernization but
  blocked by a tooling permission.
- **Concurrency story.** `node:sqlite` is synchronous, so the async API never yields
  mid-operation. Document what that means for a server using this under load, and
  whether a file-backed database is safe across multiple processes (WAL + `busy_timeout`).
